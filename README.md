# Orbit Wars - Kaggle Competition

## 概要
Kaggleのシミュレーションコンペ「Orbit Wars」への参加記録です。
100×100の宇宙空間で惑星を征服し、500ターン後の船の総数を競います。

テーブルデータコンペしか経験がない状態から、
ルールベースエージェントの開発に初挑戦しました。

## 結果
- 参加時期：2026年5月
- 最終スコア：600前後
- 使用手法：ルールベースエージェント

## 環境
- Python 3.12
- kaggle-environments
- Kaggle Notebook（GPU T4×2）

## 1日目

### v1（スコア約400）：距離だけで狙う
一番近い惑星に艦隊を送るだけのシンプルな実装。

**問題点**
- 距離だけで判断するため生産力の低い惑星を狙いがち

### v2（スコア465）：距離÷生産力で優先度をつける
`距離 ÷ (生産力 + 1)` でスコア化し、
近くて生産力が高い惑星を優先するようにした。

**改善点**
- 守備用に最低10隻（GUARD）を残すようにした

### v3：静止惑星のみを狙う
対戦画面を観察して「艦隊が惑星をすり抜けている」ことを発見。
自転惑星を狙うと到着時に惑星が移動してしまうことが原因と特定。
静止惑星のみに絞ることで艦隊の無駄をなくした。

**発見**：ローカルテストでv2に10/10勝利

**問題点**
- 自転惑星（内側の生産力が高い惑星）をほとんど取れない
- 実際の対戦では内側を先に取られて負けることが多い

### v4（スコア650前後）：少数艦隊戦略
2つの改善を組み合わせてスコアアップ。

**改善点：送る船数を20→8に削減**
少ない船で素早く多くの惑星を確保する戦略が有効と判明。
惑星を多く持つほど毎ターンの生産量が増えるため、
速く広げる方が大艦隊で攻めるより強い。

**発見**：対戦画面の観察から
「たまたま当たった移動惑星を取った方が勝つ」ことを発見。
艦隊を常に飛ばし続けることが重要と気づいた。

## 学んだこと

## 2日目の試行錯誤

### 発見：ローカルテストとSubmitスコアが逆転することがある
v4がローカルでv3に勝っても、実際のSubmitスコアはv3の方が高かった。
ローカルテストは同じ相手との対戦のため偏りが出る。
Kaggleのスコアの方が実力を正確に反映している。

### v5a（現在の最強版）：上位コードのスコア計算を取り込む
上位参加者のNotebookを分析して、スコア計算の考え方を参考にした。

**改善前（v4）のスコア計算：**
距離 ÷ (生産力 + 1)

**改善後（v5a）のスコア計算：**
(100 - 距離)
\+ (30 × 生産力)         # 生産力が高い惑星を強く優先
\+ (20 × 敵惑星ボーナス) # 敵惑星を取ると自分が得して敵が損する
\- (0.7 × 必要な船数)    # 占領コストが高い惑星を避ける
\- (2 × 到着時間)        # 到着までに敵が増産する時間を考慮

**ポイント：到着時間（eta）を考慮するようにした**
遠い惑星は到着までに敵が船を増産するため、
必要な船数が増える。この分をスコアに反映した。

**係数チューニングの結果：**
| パターン | 生産力 | 敵ボーナス | ships影響 | vs v4 |
|---|---|---|---|---|
| A（採用）| 30 | 20 | 0.7 | 7/10 |
| B（距離重視）| 15 | 10 | 0.7 | 4/10 |
| C（到着時間重視）| 15 | 10 | 0.7 | 6/10 |
| A2（生産力50）| 50 | 20 | 0.7 | 1/10 |
| A3（敵ボーナス40）| 30 | 40 | 0.7 | 3/10 |
| A4（ships影響0.3）| 30 | 20 | 0.3 | 5/10 |

→ v5a（生産力30、敵ボーナス20、ships0.7）が最強と判明

### v6（v5aより強い）：到着までの増産を考慮した必要船数計算
**改善点：敵が到着までに増産する分をships_neededに加算**

今まで：
ships_needed = 相手の船数 + 1

改善後：
eta = 距離 ÷ 艦隊速度（到着までのターン数）
extra = eta × 相手の生産力（到着までに増産される船数）
ships_needed = 相手の船数 + 1 + extra

結果：v5aに7/10勝利

---

### v7（v6より強い）：彗星を攻撃対象から除外
**改善点：comet_planet_idsで彗星を除外**

彗星は動くため狙っても艦隊がすり抜けやすい。
上位コードも彗星を除外していたため取り込んだ。
彗星しか残っていない場合は例外的に彗星も対象にする。

結果：v6に7/10勝利 → Submitへ

### 試したが効果がなかった改善（v8系）

**ships数の多い惑星から優先攻撃（v8）**
- 上位コードのソート処理を取り込んだ
- 結果：v7に1/10で大幅悪化
- 原因：船が多い惑星＝重要拠点の守りが薄くなった

**太陽衝突判定の改善**
- 上位コードのcollides関数を取り込んだ
- 太陽を通るルートのスコアを200下げた
- 結果：v7に1/10で大幅悪化
- 原因：攻撃できる惑星が減って手が止まりやすくなった

**二重攻撃防止（v8）**
- 既に十分な艦隊が向かっている惑星の優先度を下げた
- 結果：v7に2/10で悪化
- 原因：攻撃対象が絞られすぎて別の惑星を取れなくなった

### v9：fleet_trajectories（艦隊追跡）の実装

上位コードの核心部分を取り込んだ。
グローバル変数で過去に送った艦隊を記録し、
同じ惑星に二重に艦隊を送る無駄をなくした。

**仕組み：**
```python
# 艦隊ログを記録
_fleet_log.append({
    "target_id": nearest.id,
    "ships": ships_needed,
    "arrive_tick": arrive_tick
})

# 既に十分な艦隊が向かっている惑星はスキップ
already = en_route.get(t.id, 0)
if already >= t.ships + 1:
    return 999999  # 最低優先度
```

**効果：**
- v7に7/10で勝利

### v10：自転惑星の予測精度向上

小さい惑星を狙う際に惑星の中心を狙いきれておらず、送った戦艦が無駄になることがあった。そのためv9の自転惑星の予測を大幅に改善した。

**v9との違い：**
```python
# v9：固定の速度3で計算
turns = int(dist / 3) + 1
fut_angle = cur_angle + angular_velocity * turns

# v10：実際のETAで正確に計算
turns = max(1, round(eta))           # 実際の到着ターン数
lead = angular_velocity * (turns - 0.5)  # 微調整で命中率アップ
future_angle = cur_angle + lead
aim_r = max(0, r - nearest.radius * 0.25)  # 少し内側を狙う
```

**効果：**
- v9に8/10で勝利
- randomに10/10勝利
